# Проектирование базы данных
## Учебный проект (MSSQL)

---

## 1. Тема проекта
**Система мониторинга серверов и сервисов**

---

## 2. Цель работы
Разработать концептуальную и логическую модель базы данных для выбранной предметной области, привести модель к третьей нормальной форме (3НФ) и подготовить диаграмму БД с описанием.

---

## 3. Предметная область
**Описание (для сдачи):**

В рамках проекта рассматривается предметная область мониторинга IT‑инфраструктуры организации.

Система предназначена для хранения и обработки информации о:
- серверах инфраструктуры;
- операционных системах;
- сервисах, работающих на серверах;
- метриках состояния (нагрузка CPU, RAM, дисков и т.д.);
- инцидентах;
- уведомлениях;
- пользователях системы и их ролях.

Система используется администраторами и инженерами для контроля состояния инфраструктуры, анализа метрик и реагирования на инциденты.

---

## 4. Концептуальная модель (сущности)

В предметной области выделены следующие сущности:
- Roles (роли)
- Users (пользователи)
- OperatingSystems (операционные системы)
- Servers (серверы)
- ServiceTypes (типы сервисов)
- Services (сервисы)
- MetricTypes (типы метрик)
- ServerMetrics (метрики серверов)
- Statuses (статусы)
- Incidents (инциденты)
- NotificationTypes (типы уведомлений)
- Notifications (уведомления)

- Roles (1) → (M) Users
- Users (1) → (M) Servers
- OperatingSystems (1) → (M) Servers
- Servers (1) → (M) Services
- ServiceTypes (1) → (M) Services
- Servers (1) → (M) ServerMetrics
- MetricTypes (1) → (M) ServerMetrics
- Statuses (1) → (M) Incidents
- Servers (1) → (M) Incidents
- Services (1) → (M) Incidents
- Incidents (1) → (M) Notifications
- Users (1) → (M) Notifications
- NotificationTypes (1) → (M) Notifications

## 7. Описание связей между сущностями (с объяснением)

В данной базе данных используются связи типа «один ко многим» (1:M), что соответствует логике предметной области.

### Roles → Users (1:M)
Одна роль может быть назначена многим пользователям. Например, роль DBA может принадлежать нескольким сотрудникам.

### Users → Servers (1:M)
Один пользователь может быть ответственным за несколько серверов. Это позволяет закрепить администратора за группой серверов.

### OperatingSystems → Servers (1:M)
Одна операционная система может быть установлена на многих серверах. Это исключает дублирование данных об ОС.

### Servers → Services (1:M)
На одном сервере может быть запущено несколько сервисов (например, PostgreSQL, Nginx, Redis).

### ServiceTypes → Services (1:M)
Один тип сервиса может использоваться в нескольких записях. Например, тип PostgreSQL может относиться к разным серверам.

### Servers → ServerMetrics (1:M)
С одного сервера регулярно собирается множество метрик (CPU, RAM и т.д.). Каждая запись — отдельный момент времени.

### MetricTypes → ServerMetrics (1:M)
Один тип метрики (например, CPU) используется во множестве измерений.

### Statuses → Incidents (1:M)
Один статус (например, Critical) может быть назначен многим инцидентам.

### Servers → Incidents (1:M)
На одном сервере может возникнуть множество инцидентов.

### Services → Incidents (1:M)
Инциденты могут быть связаны с конкретным сервисом, работающим на сервере.

### Incidents → Notifications (1:M)
По одному инциденту может быть отправлено несколько уведомлений (например, разным пользователям или через разные каналы).

### Users → Notifications (1:M)
Один пользователь может получать множество уведомлений.

### NotificationTypes → Notifications (1:M)
Один тип уведомления (например, Email или Telegram) может использоваться во многих записях уведомлений.

## 8. Нормализация базы данных

База данных приведена к третьей нормальной форме (3НФ).

### 1НФ
- все атрибуты атомарны;
- отсутствуют повторяющиеся группы;
- каждая запись уникальна.

### 2НФ
- все неключевые атрибуты зависят от полного первичного ключа;
- отсутствуют частичные зависимости.

### 3НФ
- отсутствуют транзитивные зависимости;
- справочные данные вынесены в отдельные таблицы.

**Примеры нормализации:**
- роли вынесены в таблицу Roles;
- операционные системы вынесены в таблицу OperatingSystems;
- типы сервисов вынесены в таблицу ServiceTypes;
- типы метрик вынесены в таблицу MetricTypes;
- статусы вынесены в таблицу Statuses;
- типы уведомлений вынесены в таблицу NotificationTypes.

**Вывод:**
Нормализация уменьшает избыточность данных и предотвращает аномалии вставки, обновления и удаления.

---

## 9. Обоснование проектных решений

- Использованы справочные таблицы для устранения дублирования данных;
- Применены внешние ключи для обеспечения ссылочной целостности;
- Использованы связи типа 1:M;
- Nullable поле ServiceID в Incidents позволяет фиксировать инциденты как по серверу, так и по сервису.
