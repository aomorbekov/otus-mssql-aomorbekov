# Проект базы данных: система мониторинга серверов и сервисов

## 1. Название проекта
**Система мониторинга серверов и сервисов**

---

## 2. Цель проекта
Спроектировать базу данных для системы мониторинга IT-инфраструктуры, которая позволяет хранить информацию о серверах, установленных на них сервисах, собираемых метриках, инцидентах и уведомлениях ответственным пользователям.

---

## 3. Описание предметной области
В рамках проекта рассматривается предметная область мониторинга IT-инфраструктуры организации.

В системе необходимо хранить данные о:
- серверах;
- операционных системах, установленных на серверах;
- сервисах, работающих на серверах;
- метриках состояния серверов;
- инцидентах, возникающих при отклонении показателей от нормы;
- уведомлениях, отправляемых ответственным сотрудникам;
- пользователях системы и их ролях.

Такая база данных может использоваться администраторами, DevOps-инженерами и DBA для централизованного контроля состояния инфраструктуры и фиксации событий мониторинга.

---

## 4. Бизнес-идея проекта
Проект предназначен для хранения и структурирования информации о состоянии серверной инфраструктуры.

Система должна позволять:
- учитывать серверы и их параметры;
- отслеживать установленные сервисы;
- хранить историю метрик;
- регистрировать инциденты;
- фиксировать факт отправки уведомлений ответственным лицам.

Практическая ценность проекта заключается в том, что подобная база данных может лежать в основе системы мониторинга и эксплуатации IT-инфраструктуры.

---

## 5. Основные задачи базы данных
1. Хранение справочной информации о ролях, операционных системах, статусах, типах сервисов, типах метрик и типах уведомлений.
2. Хранение информации о пользователях системы.
3. Хранение информации о серверах и связанных с ними сервисах.
4. Хранение истории метрик по серверам.
5. Хранение информации об инцидентах.
6. Хранение информации об уведомлениях по инцидентам.

---

- Roles

- Servers
- ServiceTypes

## 7. Описание сущностей и их назначения

### 7.1. Roles
Справочник ролей пользователей системы.

**Примеры значений:** DBA, DevOps, Admin.

### 7.2. Users
Пользователи системы, ответственные за серверы и получающие уведомления.

### 7.3. OperatingSystems
Справочник операционных систем, установленных на серверах.

### 7.4. Servers
Информация о серверах инфраструктуры.

### 7.5. ServiceTypes
Справочник типов сервисов.

### 7.6. Services
Сервисы, работающие на конкретных серверах.

### 7.7. MetricTypes
Справочник типов метрик.

### 7.8. ServerMetrics
История собранных метрик серверов.

### 7.9. Statuses
Справочник статусов инцидентов.

### 7.10. Incidents
Инциденты, связанные с сервером или сервисом.

### 7.11. NotificationTypes
Справочник типов уведомлений.

### 7.12. Notifications
Журнал отправленных уведомлений по инцидентам.

---

## 8. Связи между сущностями
- Одна роль может принадлежать многим пользователям.
- Одна операционная система может быть установлена на многих серверах.
- Один пользователь может быть ответственным за несколько серверов.
- Один сервер может содержать несколько сервисов.
- Один тип сервиса может относиться к нескольким сервисам.
- Один сервер может иметь множество записей метрик.
- Один тип метрики может использоваться во множестве записей метрик.
- Один статус может быть назначен многим инцидентам.
- Один сервер может иметь много инцидентов.
- Один сервис может иметь много инцидентов.
- Один инцидент может иметь много уведомлений.
- Один пользователь может получить много уведомлений.
- Один тип уведомления может использоваться во многих уведомлениях.

## 10. Нормализация базы данных
База данных приведена к **третьей нормальной форме (3НФ)**.

### 10.1. Первая нормальная форма (1НФ)
- все таблицы содержат атомарные значения;
- отсутствуют повторяющиеся группы атрибутов;
- каждая строка таблицы уникальна.

### 10.2. Вторая нормальная форма (2НФ)
- все неключевые атрибуты зависят от полного первичного ключа;
- частичные зависимости отсутствуют.

### 10.3. Третья нормальная форма (3НФ)
- отсутствуют транзитивные зависимости;
- повторяющиеся справочные данные вынесены в отдельные таблицы.

**Примеры нормализации в проекте:**
- роли вынесены в таблицу `Roles`;
- операционные системы вынесены в таблицу `OperatingSystems`;
- типы сервисов вынесены в таблицу `ServiceTypes`;
- типы метрик вынесены в таблицу `MetricTypes`;
- статусы вынесены в таблицу `Statuses`;
- типы уведомлений вынесены в таблицу `NotificationTypes`.

**Вывод:**
Нормализация уменьшает дублирование данных, упрощает обновление записей и повышает целостность базы данных.

## 11. Предполагаемая структура таблиц

### Roles
- RoleID — PK
- RoleName

### Users
- UserID — PK
- FullName
- Email
- Login
- RoleID — FK -> Roles.RoleID

### OperatingSystems
- OSID — PK
- OSName
- OSVersion

### Servers
- ServerID — PK
- HostName
- IPAddress
- Environment
- OSID — FK -> OperatingSystems.OSID
- ResponsibleUserID — FK -> Users.UserID

### ServiceTypes
- ServiceTypeID — PK
- ServiceTypeName

### Services
- ServiceID — PK
- ServerID — FK -> Servers.ServerID
- ServiceTypeID — FK -> ServiceTypes.ServiceTypeID
- ServiceName
- Port
- IsActive

### MetricTypes
- MetricTypeID — PK
- MetricName
- Unit

### ServerMetrics
- MetricID — PK
- ServerID — FK -> Servers.ServerID
- MetricTypeID — FK -> MetricTypes.MetricTypeID
- MetricValue
- CollectedAt

### Statuses
- StatusID — PK
- StatusName

### Incidents
- IncidentID — PK
- ServerID — FK -> Servers.ServerID
- ServiceID — FK -> Services.ServiceID
- StatusID — FK -> Statuses.StatusID
- Title
- Description
- CreatedAt
- ResolvedAt

### NotificationTypes
- NotificationTypeID — PK
- TypeName

### Notifications
- NotificationID — PK
- IncidentID — FK -> Incidents.IncidentID
- UserID — FK -> Users.UserID
- NotificationTypeID — FK -> NotificationTypes.NotificationTypeID
- SentAt
- DeliveryStatus

Здесь можно указать основные ограничения, которые будут реализованы в БД.
