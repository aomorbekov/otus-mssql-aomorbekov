# Проектирование базы данных
## Учебный проект (MSSQL)

---

## 1. Тема проекта
**Система мониторинга серверов и сервисов**

---

## 2. Цель работы
Сделать модель базы данных для системы мониторинга: описать предметную область, выделить сущности, связи и привести всё к 3НФ.

---

## 3. Предметная область
В работе рассматривается мониторинг IT‑инфраструктуры.

Нужно хранить данные о:
- серверах;
- операционных системах;
- сервисах на серверах;
- метриках (CPU, RAM и т.п.);
- инцидентах;
- уведомлениях;
- пользователях и их ролях.

Система нужна, чтобы следить за состоянием серверов и быстро реагировать на проблемы.

---

## 4. Сущности
Используются следующие сущности:
- Roles
- Users
- OperatingSystems
- Servers
- ServiceTypes
- Services
- MetricTypes
- ServerMetrics
- Statuses
- Incidents
- NotificationTypes
- Notifications

### Roles
Хранит роли пользователей в системе.

Атрибуты:
- **RoleID** — первичный ключ (PK)
- **RoleName** — название роли

### Users
Хранит пользователей системы.

Атрибуты:
- **UserID** — первичный ключ (PK)
- **FullName** — ФИО пользователя
- **Email** — электронная почта
- **Login** — логин
- **RoleID** — внешний ключ (FK) на таблицу `Roles`

### OperatingSystems
Хранит список операционных систем.

Атрибуты:
- **OSID** — первичный ключ (PK)
- **OSName** — название ОС
- **OSVersion** — версия ОС

### Servers
Хранит информацию о серверах.

Атрибуты:
- **ServerID** — первичный ключ (PK)
- **HostName** — имя сервера
- **IPAddress** — IP-адрес
- **Environment** — окружение (prod, test, dev)
- **OSID** — внешний ключ (FK) на таблицу `OperatingSystems`
- **ResponsibleUserID** — внешний ключ (FK) на таблицу `Users`

### ServiceTypes
Хранит типы сервисов.

Атрибуты:
- **ServiceTypeID** — первичный ключ (PK)
- **ServiceTypeName** — название типа сервиса

### Services
Хранит сервисы, работающие на серверах.

Атрибуты:
- **ServiceID** — первичный ключ (PK)
- **ServerID** — внешний ключ (FK) на таблицу `Servers`
- **ServiceTypeID** — внешний ключ (FK) на таблицу `ServiceTypes`
- **ServiceName** — название сервиса
- **Port** — порт
- **IsActive** — признак активности сервиса

### MetricTypes
Хранит типы метрик.

Атрибуты:
- **MetricTypeID** — первичный ключ (PK)
- **MetricName** — название метрики
- **Unit** — единица измерения

### ServerMetrics
Хранит значения метрик серверов.

Атрибуты:
- **MetricID** — первичный ключ (PK)
- **ServerID** — внешний ключ (FK) на таблицу `Servers`
- **MetricTypeID** — внешний ключ (FK) на таблицу `MetricTypes`
- **MetricValue** — значение метрики
- **CollectedAt** — дата и время сбора

### Statuses
Хранит статусы инцидентов.

Атрибуты:
- **StatusID** — первичный ключ (PK)
- **StatusName** — название статуса

### Incidents
Хранит информацию об инцидентах.

Атрибуты:
- **IncidentID** — первичный ключ (PK)
- **ServerID** — внешний ключ (FK) на таблицу `Servers`
- **ServiceID** — внешний ключ (FK) на таблицу `Services`, может быть `NULL`
- **StatusID** — внешний ключ (FK) на таблицу `Statuses`
- **Title** — заголовок инцидента
- **Description** — описание инцидента
- **CreatedAt** — дата создания
- **ResolvedAt** — дата закрытия

### NotificationTypes
Хранит типы уведомлений.

Атрибуты:
- **NotificationTypeID** — первичный ключ (PK)
- **TypeName** — название типа уведомления

### Notifications
Хранит отправленные уведомления.

Атрибуты:
- **NotificationID** — первичный ключ (PK)
- **IncidentID** — внешний ключ (FK) на таблицу `Incidents`
- **UserID** — внешний ключ (FK) на таблицу `Users`
- **NotificationTypeID** — внешний ключ (FK) на таблицу `NotificationTypes`
- **SentAt** — дата и время отправки
- **DeliveryStatus** — статус доставки

## 5. Кратко по таблицам

**Roles** — таблица со списком ролей пользователей. Нужна для того, чтобы не хранить название роли текстом в каждой записи пользователя.

**Users** — таблица пользователей системы. Здесь хранятся данные о людях, которые работают с системой, получают уведомления и могут быть ответственными за серверы.

**OperatingSystems** — таблица операционных систем. Позволяет хранить список используемых ОС отдельно и не дублировать их в таблице серверов.

**Servers** — основная таблица с серверами. Здесь хранится имя сервера, IP-адрес, окружение и ссылки на ОС и ответственного пользователя.

**ServiceTypes** — таблица типов сервисов. Нужна для хранения общего списка типов, например PostgreSQL, Nginx, Redis.

**Services** — таблица сервисов, которые работают на конкретных серверах. Показывает, какой сервис на каком сервере установлен, какой у него тип, порт и активен ли он.

**MetricTypes** — таблица типов метрик. Например, CPU, RAM, Disk Usage. Это позволяет не повторять одинаковые названия метрик во многих строках.

**ServerMetrics** — таблица со значениями метрик. Здесь хранится история измерений по серверам с указанием времени сбора.

**Statuses** — таблица статусов инцидентов. Например, New, In Progress, Resolved, Critical.

**Incidents** — таблица инцидентов. Хранит информацию о проблемах, которые возникли на сервере или в сервисе.

**NotificationTypes** — таблица типов уведомлений. Например, Email, Telegram, SMS.

**Notifications** — таблица отправленных уведомлений. Хранит, по какому инциденту, кому и каким способом было отправлено уведомление.

## 6. Связи (простыми словами)

- У одной роли может быть много пользователей
- Один пользователь может отвечать за несколько серверов
- На сервере может быть много сервисов
- У сервера много метрик
- Один тип метрики используется в разных записях
- У сервера могут быть инциденты
- Инциденты могут относиться к сервису (или только к серверу)
- У одного инцидента может быть несколько уведомлений
- Один пользователь может получить много уведомлений

## 8. Нормализация
База данных приведена к 3НФ.

- 1НФ — все поля атомарные
- 2НФ — нет частичных зависимостей
- 3НФ — нет лишних зависимостей между полями

Справочники вынесены отдельно (роли, статусы, типы и т.д.), чтобы не было дублирования.

---

## 9. Почему так сделал

- Использовал отдельные таблицы для справочников
- Сделал связи через внешние ключи
- Взял связи 1 ко многим — это логично для такой системы
- В Incidents поле ServiceID может быть NULL — потому что инцидент может быть не только по сервису
